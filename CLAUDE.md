# Mindaptive Responder

A self-hosted dashboard that helps Andrew (Mindaptive.ai) follow up with Smartlead leads: it surfaces leads with no contact in 3+ days, drafts a follow-up with Claude on click (single or bulk), and lets him review/edit/send from a browser. Lead replies still auto-draft immediately so hot leads get a fast response.

## Where to make changes for better messages (start here)

These are plain markdown files, not code ‚Äî edit them directly, no Python knowledge needed:

- **`prompts/system.md`** ‚Äî the SDR system prompt. Role, tone rules (Hormozi-style: brief, peer-to-peer, no pricing over email), the website-diagnostic framework, objection handling, output format, and the follow-up examples (section 12). This is the first place to change *how* the AI writes.
- **`knowledge/solutions-catalog.md`** ‚Äî every product/solution Mindaptive sells, organized by the RACE framework, with per-niche packages. Add or update offerings here.
- **`knowledge/vsl-transcript.md`** ‚Äî the AI-secretary VSL transcript; referenced when deciding whether to send the video link.
- **`knowledge/hormozi-communication-style.md`** ‚Äî sales philosophy/scripts reference (objection handling, follow-up cadence patterns).

`app/drafter.py` concatenates `prompts/system.md` + all of `knowledge/*.md` + a short "output contract" addendum (instructs the model to wrap its triage/draft/translation in `<triage>`/`<draft_original>`/`<draft_english>` tags so the app can parse them) into the system prompt sent to Claude. Adding a new `.md` file to `knowledge/` automatically includes it ‚Äî no code change needed.

## How a message actually gets generated

1. **Daily scan** (`app/scheduler.py: run_daily_scan`) ‚Äî cron job (`DAILY_SCAN_HOUR_UTC`), Smartlead API only, no Claude. Also triggerable on demand from the dashboard ("Rescan now" button ‚Üí `scheduler.trigger_scan_in_background`, lock-protected so it can't stack). For every "Interested" lead it checks the thread and decides: due for a follow-up (per the `FOLLOWUP_WAIT_DAYS` cadence list, under the follow-up cap ‚Äî plus a one-shot revival touch every `REVIVE_AFTER_DAYS` after the cap) ‚Üí adds a row to the `candidates` table; lead's message unanswered ‚Üí auto-drafts a reply immediately (fast response to hot leads); otherwise ‚Üí nothing. Leads in Smartlead's **"Meeting-Booked" category** (matched case/punctuation-insensitively, `MEETING_BOOKED_CATEGORY_NAME`) get frozen instead: status `booked`, open drafts ‚Üí stale, candidates dismissed, `booked_at` recorded (`db.mark_lead_booked`) ‚Äî and un-frozen automatically if the category later moves back to Interested. Booked leads stay visible in the inbox with a green "Meeting booked ‚úÖ" badge.
2. **Overnight batch pre-generation** (`app/batch_gen.py`, `AUTO_GENERATE_FOLLOWUPS=true` ‚Äî approved on-by-default 2026-07-19) ‚Äî right after the scan, every open follow-up candidate whose lead already has `research_summary` is submitted as ONE Anthropic **Batch API** job (50% token cost; no web tools, since a batch result can't continue a pause_turn loop ‚Äî leads without research stay open for interactive Generate). A 5-minute poll job consumes finished batches: each result is race-checked against a fresh thread, then stored through the same `pipeline.store_draft_result` as interactive drafts. Sending still always requires Andrew's click; only if `AUTO_SEND_FOLLOWUPS` is also on do these drafts go straight to `scheduled` for the lead's next weekday morning.
3. **Generate / Bulk generate** (`app/candidates.py`) ‚Äî user-triggered fallback for anything not pre-generated. Re-fetches the thread fresh (race-check: skip if the lead already replied), then calls `app/pipeline.py: create_draft` ‚Üí `app/drafter.py: generate_draft`, which calls Claude with web search/fetch tools enabled so it can research the lead's website (capped: `max_uses: 4` each, fetches ‚â§10k tokens, and the growing tool-loop conversation carries a `cache_control` breakpoint so pause_turn continuations read prior turns at ~0.1√ó). Bulk generate runs in a background thread so it doesn't block on Cloudflare's ~100s tunnel timeout. The final follow-up before the cap is steered into the Section-12 "closing this file" breakup pattern; post-cap revival drafts get fresh-angle steering (`followup_stage` in `pipeline.create_draft`).
4. **Signature** ‚Äî `pipeline.create_draft` looks up which mailbox sent the thread's last message (`detector.last_sender_email`) and stores that persona's HTML signature (`app/signatures.py`, files in `signatures/`) in the `drafts.signature_html` column. The signature is **kept out of `body_html`** ‚Äî `body_html` holds the message body only. The signature is shown in the dashboard as a separate read-only preview under the editor (`app/static/app.js: renderDraftSection`, visible in both the Original and English tabs) and is appended **once, unchanged**, at send time in `scheduler.compose_send_body`. This deliberately keeps the signature out of the English translate/localize round-trip: it used to be baked into `body_html`, which meant the English tab translated the signature and `api_draft_localize` then appended a second copy ‚Äî producing two signatures, one mangled. Because `body_html` is now guaranteed signature-free, the append in `compose_send_body` is a plain, deterministic append (guarded by a substring check only to protect any pre-change draft that still has it baked in). Persona is detected from Smartlead's own `from_name` on each of the 90+ rotating sending accounts (`GET /email-accounts`), not a hand-maintained email list ‚Äî see `PERSONA_FILES` in `signatures.py` if a new persona/name is added.
5. **Lead research** ‚Äî the same Claude call that drafts the message also researches the lead's website (per the Website Diagnostic Framework in `prompts/system.md`) and returns a `<lead_research>` block (parsed into `DraftResult.lead_research`). `pipeline.create_draft` saves this to `leads_state.research_summary`/`researched_at` and passes it back in as `prior_research` on every later generation for that lead (regenerate, next follow-up, etc.) ‚Äî the model is instructed to reuse it and skip re-searching unless it looks stale or a steering note asks for something new (tools stay available, it's an instruction not a hard cutoff). Shown in the dashboard as the "About this lead" panel next to the thread (`app/static/app.js: renderResearchPanel`).
6. **Review** ‚Äî draft appears as an editable card (triage summary, English translation, edit body incl. signature, thread history). Actions: Send now, Schedule, Regenerate (with an optional steering note), Skip, Stop following up this lead. Timing rule by kind: **replies send now** (speed-to-lead), **follow-ups schedule** ‚Äî the Schedule picker pre-fills with the lead's next weekday ~9am in the campaign timezone (`thread_utils.next_morning_send_utc` off `leads_state.timezone_guess`, surfaced as `draft.suggested_schedule_at`); the client converts the picked local time to UTC before posting (`scheduleDraft`), so schedules fire when intended.
7. **Send** ‚Äî `app/smartlead.py: reply_to_thread`. POST `/campaigns/{id}/reply-email-thread` body is `email_stats_id` (required ‚Äî Smartlead's internal id, captured as `NormalizedMessage.stats_id`/`drafts.reply_stats_id`, distinct from `reply_message_id` the RFC822 header value), `email_body` (required), `reply_message_id`/`reply_email_time` (optional but sent for threading). **`lead_id` is not a valid field on this endpoint and gets rejected** ("lead_id is not allowed") ‚Äî confirmed against Smartlead's own reference and a real 400 in production; don't add it back. On success the endpoint returns **plain text** (`"Email added to the queue, will be sent out soon!"`, `content-type: text/html`), not JSON ‚Äî `smartlead._request` checks `content-type` before calling `.json()` for exactly this reason. `_send_due_draft` re-fetches the thread fresh right before sending (for the race-check) and uses that fresh `last.stats_id` rather than the value stored on the draft, so older drafts (predating the `reply_stats_id` column) don't send a NULL. Re-checks the thread once more immediately before sending ‚Äî for a reply/autoreply draft this only aborts if the lead has a *newer* reply than the one the draft addresses (comparing `message_id`), not merely "last message is a reply", since that's always true for a reply-kind draft by definition.

8. **Images in a draft** ‚Äî paste a screenshot straight from the clipboard, drag a file onto the editor, or use the toolbar's "Image" button: the file is POSTed to `/api/uploads` (authed), sniffed by magic bytes (PNG/JPG/GIF/WEBP only, ‚â§8 MB, `app/uploads.py`) and written to `UPLOAD_DIR` (`/data/uploads`, same Docker volume as the DB). It's inserted as an absolute `PUBLIC_BASE_URL/i/<random>.<ext>` URL. **`GET /i/{name}` is deliberately the one unauthenticated read route** ‚Äî the recipient's mail client fetches it with no session; the random filename is the capability, and the name is regex-validated before touching the filesystem. Pasting a bare image URL (the old imgur route) also becomes a real `<img>` rather than link text. Clicking any image in the editor opens a floating resize bar (S/M/L/Full presets + slider) which writes the pixel size to **both** the `width` attribute and inline `style` ‚Äî Outlook ignores CSS width on images ‚Äî with 100% capped at 600px. The selection outline is a CSS class stripped in `editorSerialize` so it never ships in `body_html`.

9. **Stats** ‚Äî dashboard "Stats" tab (`GET /api/metrics`, rendered by `renderStats` in `app/static/app.js`): meetings booked (from `leads_state.booked_at`), sends by kind, follow-up‚Üíreply rate (proxy: a reply-kind draft created after the follow-up's `sent_at`), avg lead-reply‚Üíour-send hours, current pipeline counts, and drafts per model (`drafts.model`, NULL for manual/template drafts). All derived from existing tables ‚Äî no separate event tracking.

## Key config (`.env`, see `.env.example`)

- `FOLLOWUP_WAIT_DAYS` / `MAX_FOLLOWUPS` ‚Äî cadence and cap. `FOLLOWUP_WAIT_DAYS` accepts a comma list (`3,4,6,8` ‚Äî wait before follow-up #N, last value repeats) or a single number (old fixed cadence).
- `REVIVE_AFTER_DAYS` ‚Äî after the cap, resurface a cold lead for one revival touch every N days of silence (0 disables).
- `MEETING_BOOKED_CATEGORY_NAME` ‚Äî Smartlead category that freezes outreach and counts as a booked meeting (Stats view).
- `DRY_RUN=true` ‚Äî pipeline runs fully but the actual Smartlead send is skipped and logged instead. Keep this on for any testing.
- `AUTO_GENERATE_FOLLOWUPS` ‚Äî overnight Batch API pre-generation of due follow-up drafts (on by default; generation only, never sends).
- `AUTO_SEND_FOLLOWUPS` ‚Äî autonomous mode: batch-pre-generated follow-ups go straight to `scheduled` (next lead-local weekday morning) and scheduled sends fire without review. Off by default.
- `ANTHROPIC_MODEL` ‚Äî currently `claude-sonnet-5`.

## Local dev

```
python -m venv .venv
./.venv/Scripts/pip install -r requirements.txt   # Scripts/ on Windows, bin/ on Linux/Mac
cp .env.example .env   # fill in SMARTLEAD_API_KEY, ANTHROPIC_API_KEY, APP_PASSWORD
./.venv/Scripts/python -m uvicorn app.main:app --reload --port 8080
```

Dashboard at `http://localhost:8080` (password = `APP_PASSWORD`). Keep `DRY_RUN=true` while testing ‚Äî the daily scan and generate/bulk-generate flows call real Smartlead + Anthropic APIs and will incur real Claude token cost per draft generated.

## Things not to change without a reason

- `app/db.py` schema (`leads_state`, `drafts`, `candidates`) ‚Äî other modules assume these exact columns.
- `app/smartlead.py` field-name guesses in `normalize_lead`/`normalize_message` ‚Äî verified against the real Smartlead API response shape (nested `lead{}` object, `lead_category_id` at the top level, message-history fields `type`/`time`/`message_id`/`email_body`).
- The webhook payload shape in `app/webhook.py` (`campaign_id`, `sl_email_lead_id`, `reply_message.text`, `to_name`, `to_email`) ‚Äî confirmed from the existing n8n workflow, not guessed.
- `signatures/andrew.html` / `signatures/mia.html` ‚Äî real signatures pulled from Smartlead's actual sending accounts, HTML/inline-styled, not meant to be hand-edited casually. `PERSONA_FILES` in `app/signatures.py` maps a Smartlead `from_name` to one of these files.
- Smartlead API response shapes are inconsistent per endpoint (some wrap in `{"data": [...]}`, others return a bare list, `message-history` wraps in `{"history": [...]}`, `reply-email-thread` returns plain text on success) ‚Äî always check `isinstance(data, list)` *before* falling back to `.get("data")`, not after (a real bug fixed in `smartlead.py`'s pagination helpers), and `smartlead._request` checks `content-type` before assuming JSON.
