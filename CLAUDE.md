# Mindaptive Responder

A self-hosted dashboard that helps Andrew (Mindaptive.ai) follow up with Smartlead leads: it surfaces leads with no contact in 3+ days, drafts a follow-up with Claude on click (single or bulk), and lets him review/edit/send from a browser. Lead replies still auto-draft immediately so hot leads get a fast response.

## Where to make changes for better messages (start here)

These are plain markdown files, not code — edit them directly, no Python knowledge needed:

- **`prompts/system.md`** — the SDR system prompt. Role, tone rules (Hormozi-style: brief, peer-to-peer, no pricing over email), the website-diagnostic framework, objection handling, output format, and the follow-up examples (section 12). This is the first place to change *how* the AI writes.
- **`knowledge/solutions-catalog.md`** — every product/solution Mindaptive sells, organized by the RACE framework, with per-niche packages. Add or update offerings here.
- **`knowledge/vsl-transcript.md`** — the AI-secretary VSL transcript; referenced when deciding whether to send the video link.
- **`knowledge/hormozi-communication-style.md`** — sales philosophy/scripts reference (objection handling, follow-up cadence patterns).

`app/drafter.py` concatenates `prompts/system.md` + all of `knowledge/*.md` + a short "output contract" addendum (instructs the model to wrap its triage/draft/translation in `<triage>`/`<draft_original>`/`<draft_english>` tags so the app can parse them) into the system prompt sent to Claude. Adding a new `.md` file to `knowledge/` automatically includes it — no code change needed.

## How a message actually gets generated

1. **Daily scan** (`app/scheduler.py: run_daily_scan`) — cron job (`DAILY_SCAN_HOUR_UTC`), Smartlead API only, no Claude. Also triggerable on demand from the dashboard ("Rescan now" button → `scheduler.trigger_scan_in_background`, lock-protected so it can't stack). For every "Interested" lead it checks the thread and decides: due for a follow-up (per the `FOLLOWUP_WAIT_DAYS` cadence list, under the follow-up cap — plus a one-shot revival touch every `REVIVE_AFTER_DAYS` after the cap) → adds a row to the `candidates` table; lead's message unanswered → auto-drafts a reply immediately (fast response to hot leads); otherwise → nothing. Leads in Smartlead's **"Meeting-Booked" category** (matched case/punctuation-insensitively, `MEETING_BOOKED_CATEGORY_NAME`) get frozen instead: status `booked`, open drafts → stale, candidates dismissed, `booked_at` recorded (`db.mark_lead_booked`) — and un-frozen automatically if the category later moves back to Interested. Booked leads stay visible in the inbox with a green "Meeting booked ✅" badge.
2. **Overnight batch pre-generation** (`app/batch_gen.py`, `AUTO_GENERATE_FOLLOWUPS=true` — approved on-by-default 2026-07-19) — right after the scan, every open follow-up candidate whose lead already has `research_summary` is submitted as ONE Anthropic **Batch API** job (50% token cost; no web tools, since a batch result can't continue a pause_turn loop — leads without research stay open for interactive Generate). A 5-minute poll job consumes finished batches: each result is race-checked against a fresh thread, then stored through the same `pipeline.store_draft_result` as interactive drafts. Sending still always requires Andrew's click; only if `AUTO_SEND_FOLLOWUPS` is also on do these drafts go straight to `scheduled` for the lead's next weekday morning.
3. **Generate / Bulk generate** (`app/candidates.py`) — user-triggered fallback for anything not pre-generated. Re-fetches the thread fresh (race-check: skip if the lead already replied), then calls `app/pipeline.py: create_draft` → `app/drafter.py: generate_draft`, which calls Claude with web search/fetch tools enabled so it can research the lead's website (capped: `max_uses: 4` each, fetches ≤10k tokens, and the growing tool-loop conversation carries a `cache_control` breakpoint so pause_turn continuations read prior turns at ~0.1×). Bulk generate runs in a background thread so it doesn't block on Cloudflare's ~100s tunnel timeout. The final follow-up before the cap is steered into the Section-12 "closing this file" breakup pattern; post-cap revival drafts get fresh-angle steering (`followup_stage` in `pipeline.create_draft`).
4. **Signature** — `pipeline.create_draft` looks up which mailbox sent the thread's last message (`detector.last_sender_email`) and stores that persona's HTML signature (`app/signatures.py`, files in `signatures/`) in the `drafts.signature_html` column. The signature is **kept out of `body_html`** — `body_html` holds the message body only. The signature is shown in the dashboard as a separate read-only preview under the editor (`app/static/app.js: renderDraftSection`, visible in both the Original and English tabs) and is appended **once, unchanged**, at send time in `scheduler.compose_send_body`. This deliberately keeps the signature out of the English translate/localize round-trip: it used to be baked into `body_html`, which meant the English tab translated the signature and `api_draft_localize` then appended a second copy — producing two signatures, one mangled. Because `body_html` is now guaranteed signature-free, the append in `compose_send_body` is a plain, deterministic append (guarded by a substring check only to protect any pre-change draft that still has it baked in). Persona is detected from Smartlead's own `from_name` on each of the 90+ rotating sending accounts (`GET /email-accounts`), not a hand-maintained email list — see `PERSONA_FILES` in `signatures.py` if a new persona/name is added.
5. **Lead research** — the same Claude call that drafts the message also researches the lead's website (per the Website Diagnostic Framework in `prompts/system.md`) and returns a `<lead_research>` block (parsed into `DraftResult.lead_research`). `pipeline.create_draft` saves this to `leads_state.research_summary`/`researched_at` and passes it back in as `prior_research` on every later generation for that lead (regenerate, next follow-up, etc.) — the model is instructed to reuse it and skip re-searching unless it looks stale or a steering note asks for something new (tools stay available, it's an instruction not a hard cutoff). Shown in the dashboard as the "About this lead" panel next to the thread (`app/static/app.js: renderResearchPanel`).
6. **Review** — draft appears as an editable card (triage summary, English translation, edit body incl. signature, thread history). Every thread message shows the mailbox it actually came from (`msg-from`, fed by `from_email` in `_thread_payload`) — for a lead's reply that's frequently a real person answering outreach that went to a generic `info@`. The lead's first name is correctable in place (**✎ Rename** → `POST /api/leads/{cid}/{lid}/name`), which sets `leads_state.name_locked=1` so the next scan doesn't revert it to Smartlead's imported `first_name` (`scheduler._process_lead`). Actions: Send now, Schedule, Regenerate (with an optional steering note), Skip, Stop following up this lead. Timing rule by kind: **replies send now** (speed-to-lead), **follow-ups schedule** — the Schedule picker pre-fills with the lead's next weekday ~9am in the campaign timezone (`thread_utils.next_morning_send_utc` off `leads_state.timezone_guess`, surfaced as `draft.suggested_schedule_at`); the client converts the picked local time to UTC before posting (`scheduleDraft`), so schedules fire when intended.
7. **Recipients** — the draft card shows an editable **To** and **Cc** row directly above Send/Schedule, so nothing goes out to an address that wasn't seen first (`renderRecipients` in `app/static/app.js`, built server-side by `main._recipients_payload`). **To** defaults to the address the lead last *wrote from* (`detector.next_reply_to`), not the imported `leads_state.email` — those differ exactly in the `info@` case above. **Cc** defaults to the thread-derived list (`detector.next_reply_cc`) and is where extra recipients go. Edits are stored as `drafts.to_override` / `drafts.cc_override` and are honoured by both the immediate and the scheduled send (`_send_due_draft`); `NULL` means "never edited, use the auto-derived value", while an **empty `cc_override` is a real value** meaning the auto Cc was deliberately cleared. `main._clean_cc` normalizes the free-text boxes into bare comma-separated addresses and silently drops anything that isn't one — a malformed Cc fails the whole send.

8. **Send** — `app/smartlead.py: reply_to_thread`. POST `/campaigns/{id}/reply-email-thread` body is `email_stats_id` (required — Smartlead's internal id, captured as `NormalizedMessage.stats_id`/`drafts.reply_stats_id`, distinct from `reply_message_id` the RFC822 header value), `email_body` (required), `reply_message_id`/`reply_email_time` (optional but sent for threading), plus `to_email` and `cc` from the recipients row. **`to_email` is always sent explicitly** (verified accepted against the real API 2026-07-21): the reference says To otherwise "defaults to lead email", i.e. the *imported* address — wrong whenever a real person replied from their own mailbox. Sending it explicitly makes the address shown in the dashboard the address that actually receives the mail. The same reference also documents `bcc`, `scheduled_time`, `add_signature` and `attachments` as valid optional fields; none are used yet. **`lead_id` is not a valid field on this endpoint and gets rejected** ("lead_id is not allowed") — confirmed against Smartlead's own reference and a real 400 in production; don't add it back. On success the endpoint returns **plain text** (`"Email added to the queue, will be sent out soon!"`, `content-type: text/html`), not JSON — `smartlead._request` checks `content-type` before calling `.json()` for exactly this reason. `_send_due_draft` re-fetches the thread fresh right before sending (for the race-check) and uses that fresh `last.stats_id` rather than the value stored on the draft, so older drafts (predating the `reply_stats_id` column) don't send a NULL. Re-checks the thread once more immediately before sending — for a reply/autoreply draft this only aborts if the lead has a *newer* reply than the one the draft addresses (comparing `message_id`), not merely "last message is a reply", since that's always true for a reply-kind draft by definition.

9. **Message templates** — the canned follow-ups behind "Choose a template…" live in the `message_templates` table and are added/edited/reordered/deleted from the modal itself (`GET|POST /api/templates`, `PATCH|DELETE /api/templates/{id}`, `POST /api/templates/{id}/move`; every mutating route returns the whole fresh list so the client never reconciles by hand). They used to be a hardcoded array in `app.js`, so a wording change meant a deploy. `app/message_templates.py: DEFAULT_TEMPLATES` is **seed-only**: `db.init_db` checks whether the table existed *before* running the schema and seeds only on first creation, so templates Andrew deletes never come back on restart. `{name}`/`{company}` are stored raw and resolved client-side (`fillPlaceholders`) right before `/quick-draft`.

10. **Images in a draft** — paste a screenshot straight from the clipboard, drag a file onto the editor, or use the toolbar's "Image" button: the file is POSTed to `/api/uploads` (authed), sniffed by magic bytes (PNG/JPG/GIF/WEBP only, ≤8 MB, `app/uploads.py`) and written to `UPLOAD_DIR` (`/data/uploads`, same Docker volume as the DB). It's inserted as an absolute `PUBLIC_BASE_URL/i/<random>.<ext>` URL. **`GET /i/{name}` is deliberately the one unauthenticated read route** — the recipient's mail client fetches it with no session; the random filename is the capability, and the name is regex-validated before touching the filesystem. Pasting a bare image URL (the old imgur route) also becomes a real `<img>` rather than link text. Clicking any image in the editor opens a floating resize bar (S/M/L/Full presets + slider) which writes the pixel size to **both** the `width` attribute and inline `style` — Outlook ignores CSS width on images — with 100% capped at 600px. The selection outline is a CSS class stripped in `editorSerialize` so it never ships in `body_html`.

11. **Campaign analysis** — dashboard "Campaigns" tab: which outbound message variants actually earn replies, and how to build the next run. Two layers, both cached in the DB behind one **Analyze** click (`POST /api/campaigns/{id}/analyze` → background thread, same reason as bulk generate; the client polls `/report` while `status='running'`).
    - **Layer 1, quantitative** (`app/campaign_analytics.py`) — the key insight is that a Smartlead A/B variant is a *recipe of slots*: campaign 3640877's six step-1 variants are all `{{subjectLineN}} + {{Greeting}} + {{icebreaker*}} + {{PitchN}} + {{OfferN}} + {{CTAN}} + {{Goodbye}}`, differing only in which numbered value each picks. Slots repeat across variants (CTA1 is in both A and C), so a slot's numbers **pool every variant that uses it** — 2-3× the sample of a six-way split. `slot_role()` buckets tokens by normalized stem so `subjectLine1`, `SubjectLine1_translated` and `Icebreaker1` all land correctly. Every rate carries a **Wilson 95% interval** and a verdict (`solid_above` / `leaning_*` / `not_enough_data`); with ~40 human replies split six ways most rows are honestly `not_enough_data`, and the UI renders that neutral grey so it never reads as a result.
    - **Layer 2, qualitative** (`app/campaign_conversations.py`) — pulls the full threads of leads who answered *like people* (every category except Auto-Reply / Out Of Office / bounces; Not Interested and Do Not Contact are kept — a rejection is informative), then a map/reduce Claude pass extracts per-conversation objections, requested lead magnets, tone and triggers and synthesizes them. `first_reply_after_step` (from `email_seq_number` on our sent messages) answers "how many follow-ups still work" — on the Dental campaign step 2 outperforms step 1 3:1.
    - **Attribution is sequence-wide, not step-wise.** Only step 1 has variants, so a reply at step 2 still belongs to the variant that opened the thread. Scoring step-1 rows alone threw away two thirds of the evidence (14 human replies instead of 42).
    - **Open rate does not exist here.** Tracking is off account-wide (`DONT_EMAIL_OPEN` / `DONT_LINK_CLICK`) because pixels hurt deliverability, so `open_count` is 0 on every row. Replies are the only signal, and both prompts are explicitly forbidden from mentioning opens or clicks.
    - The variables come from Smartlead itself — `leads-export`'s `custom_fields` column is the source spreadsheet, so no CSV upload is needed. See `docs/smartlead-api.md` for the two counting traps in `/statistics` that this depends on getting right.

12. **Stats** — dashboard "Stats" tab (`GET /api/metrics`, rendered by `renderStats` in `app/static/app.js`): meetings booked (from `leads_state.booked_at`), sends by kind, follow-up→reply rate (proxy: a reply-kind draft created after the follow-up's `sent_at`), avg lead-reply→our-send hours, current pipeline counts, and drafts per model (`drafts.model`, NULL for manual/template drafts). All derived from existing tables — no separate event tracking.

## Key config (`.env`, see `.env.example`)

- `FOLLOWUP_WAIT_DAYS` / `MAX_FOLLOWUPS` — cadence and cap. `FOLLOWUP_WAIT_DAYS` accepts a comma list (`3,4,6,8` — wait before follow-up #N, last value repeats) or a single number (old fixed cadence).
- `REVIVE_AFTER_DAYS` — after the cap, resurface a cold lead for one revival touch every N days of silence (0 disables).
- `MEETING_BOOKED_CATEGORY_NAME` — Smartlead category that freezes outreach and counts as a booked meeting (Stats view).
- `DRY_RUN=true` — pipeline runs fully but the actual Smartlead send is skipped and logged instead. Keep this on for any testing.
- `AUTO_GENERATE_FOLLOWUPS` — overnight Batch API pre-generation of due follow-up drafts (on by default; generation only, never sends).
- `AUTO_SEND_FOLLOWUPS` — autonomous mode: batch-pre-generated follow-ups go straight to `scheduled` (next lead-local weekday morning) and scheduled sends fire without review. Off by default.
- `ANTHROPIC_MODEL` — currently `claude-sonnet-5`. Note adaptive thinking is on by default on this model and `max_tokens` bounds thinking **plus** visible output: `app/campaign_report.py` asks for long reports and must keep `_REPORT_MAX_TOKENS` generous (an 8k budget returned a completely empty response — all of it went to thinking).

## Local dev

```
python -m venv .venv
./.venv/Scripts/pip install -r requirements.txt   # Scripts/ on Windows, bin/ on Linux/Mac
cp .env.example .env   # fill in SMARTLEAD_API_KEY, ANTHROPIC_API_KEY, APP_PASSWORD
./.venv/Scripts/python -m uvicorn app.main:app --reload --port 8080
```

Dashboard at `http://localhost:8080` (password = `APP_PASSWORD`). Keep `DRY_RUN=true` while testing — the daily scan and generate/bulk-generate flows call real Smartlead + Anthropic APIs and will incur real Claude token cost per draft generated.

## Smartlead API documentation — what's actually available

**Read `docs/smartlead-api.md` before touching any Smartlead call.** It records
how to find a given endpoint's real schema and — more importantly — every place
the published docs are provably wrong.

The short version: the pages we depend on are mirrored into **`docs/smartlead/`**
(refresh via `python -m scripts.fetch_smartlead_docs`). For anything else,
`sitemap.xml` lists every page and **appending `.md` to a docs URL returns its
exact source** — fetch that, never the 670 KB rendered HTML. There is no
OpenAPI spec or Postman collection. The `SmartleadApi` file at the repo root is
**only an index** (a copy of `llms.txt`) with no field schemas — never treat it
as a spec.

**Ground truth is the live API, not the docs.** Verified 2026-07-21: the
documented response for `reply-email-thread` (`{"success": true, ...}`, JSON) is
not what it returns (plain text, `text/html`), and the documented response shape
for `message-history` is wrong in both the wrapper key and every field name.
Before relying on any field, probe it against the Mindaptive Jones test lead
(below) and record the result in `docs/smartlead-api.md`.

## Things not to change without a reason

- `app/db.py` schema (`leads_state`, `drafts`, `candidates`, `message_templates`, and the `campaign_*` analytics tables) — other modules assume these exact columns.
- The reply-counting rules in `app/campaign_analytics.py: _metrics` / `lead_outcomes` — `lead_category` is stamped on sends that drew no reply, and one reply appears on several step rows. Both were real bugs; only `reply_time` marks a reply, and replies are counted per unique lead.
- `app/smartlead.py` field-name guesses in `normalize_lead`/`normalize_message` — verified against the real Smartlead API response shape (nested `lead{}` object, `lead_category_id` at the top level, message-history fields `type`/`time`/`message_id`/`email_body`).
- The webhook payload shape in `app/webhook.py` (`campaign_id`, `sl_email_lead_id`, `reply_message.text`, `to_name`, `to_email`) — confirmed from the existing n8n workflow, not guessed.
- `signatures/andrew.html` / `signatures/mia.html` — real signatures pulled from Smartlead's actual sending accounts, HTML/inline-styled, not meant to be hand-edited casually. `PERSONA_FILES` in `app/signatures.py` maps a Smartlead `from_name` to one of these files.
- Smartlead API response shapes are inconsistent per endpoint (some wrap in `{"data": [...]}`, others return a bare list, `message-history` wraps in `{"history": [...]}`, `reply-email-thread` returns plain text on success) — always check `isinstance(data, list)` *before* falling back to `.get("data")`, not after (a real bug fixed in `smartlead.py`'s pagination helpers), and `smartlead._request` checks `content-type` before assuming JSON.
